# 📚 AI 长期记忆机制全景透视与 QMem 架构映射

由 QMem (PRO v2) 开发者与 AI 协同整理，系统化探究大模型长期记忆的存储、写入与召回本质。

---

## 🏗️ 第一部分：记忆存储（Memory Storage）—— 空间拓扑与分层

大模型的原生上下文（Context Window）属于“瞬时内存（RAM）”，随会话结束而清空。构建长期记忆的核心，在于将外部持久化介质与模型的认知空间进行层次化映射。

### 1. 层次化金字塔存储（Hierarchical Storage）
* **主流技术思想**：模拟计算机系统的 L1/L2/L3/Disk 缓存结构。根据信息的时效性、高频度、全局性对存储介质进行动态分层，越高的层级空间越小、离模型越近。
* **QMem 架构映射**：
  * **L1/L2（超高速缓存/动态内存）** → **Q1: Working Memory**：捕获当前会话的最核心上下文。
  * **L3（高频闪存/规约）** → **Q2: Consensus Memory & CLAUDE.md**：常驻或高频推入的全局开发规范与置顶共识。
  * **Disk（冷数据硬盘）** → **Q3 & Q4 数据库**：底层基于 SQLite3，存储结构上下文与海量的情境流水账。

### 2. 知识图谱化结构存储（Graph-based Storage）
* **主流技术思想**：人类记忆以网状概念（Concepts）和实体（Entities）相互关联，而非孤立的文本切片。使用图数据库或关联三元组（Subject-Predicate-Object）来表达逻辑链路（例如：“组件A —[重构]→ Mojo —[解决]→ GIL锁延迟”）。
* **QMem 演进方向**：可在 Q3: Structural Context 静态分析（如 `ast_parser.py`）中，提取类、函数、文件的依赖树，将其转化为结构化的实体关系关联。

## 💾 第二部分：记忆写入与固化（Memory Write & Consolidation）

如何防止外部数据库沦为“数据垃圾场”？高效的长期记忆系统必须具备提炼、压缩、更新与遗忘的能力。

### 1. 异步反射与记忆固化（Asynchronous Reflection）
* **主流技术思想**：模拟人类的睡眠记忆固化机制。系统在“清醒”（用户处于交互状态）时不做高延迟的深度处理，仅做事件日志（Event Log）的 Append-Only 写入；在系统“静默”（交互间歇、Git Commit 时）启动轻量模型进行后台异步反思，将零碎、冗长的交互历史提炼为高浓度的知识条目（Observations）。
* **QMem 架构映射**：
  * **推送模式 (Push)**：LLM 发现重要决策或 Bug 修复时，通过 `mem_save` 或 `mem_update` 触发 Upsert 写入。
  * **未来优化**：利用 Mojo 的高效多线程处理能力，可在后台挂载一个低成本的异步守护进程，对历史数据进行定期的合并与整理（Consolidation）。

### 2. 语义覆盖与遗忘机制（Semantic Overwrite & Eviction）
* **主流技术思想**：
  * **LRU（最近最少使用）**：根据记忆的激活频率动态调整权重，长久不被提及的记忆自动退化或降级。
  * **冲突覆盖（Semantic Overwrite）**：当新决策（如“切换为 SQLite”）写入时，系统自动识别并修改与其冲突的旧决策（如“使用 Redis”）为 `[Deprecated]` 状态。
* **QMem 架构映射**：
  * 目前基于 Topic Key 实现了针对指定主题的自动覆盖更新 (Upsert)。

## 🔍 第三部分：记忆检索与召回（Search & Retrieval）

普通的向量 RAG（检索增强生成）由于缺乏时序感知且容易断章取义，无法直接胜任“记忆召回”场景。前沿的记忆检索技术核心聚焦于主动权转换、时序融合与混合检索。

### 1. 认知中断与 AI 双向主动召回（Agent-Driven Proactive Retrieval）
* **主流技术思想**：打破外围系统被动塞入记忆（被动 RAG）的死结。参考操作系统虚拟内存管理（MMU）的“缺页中断”机制。大模型在思考（Thinking Process）中意识到上下文缺失时，会主动暂停回答并自我触发系统调用（如调用 `query_memory`），由外部系统（如 Mojo 侧）换入（Swap-in）新的记忆 Page 到上下文（RAM）中。
* **QMem 架构映射**：
  * 完全契合 Pull & Query 模式。在大模型需要时，利用 MCP 协议赋予其主动拉取语义与探测结构的权力，把召回的主动权交还给 AI。

### 2. 时序感知检索（Time-Aware / Temporal Retrieval）
* **主流技术思想**：让时间成为检索的第一等公民。普通 RAG 仅计算语义相似度（Cos-Similarity），导致两年前的旧决策与昨天的新决策得分一致。前沿算法引入“时序衰减权重”，优先捞出最新、离当前时机最近的记忆。
* **QMem 架构映射**：
  * QMem 独立设立了时间线概念，对后续多模态数据的时序对齐提供了天然底座。

### 3. 多模态/双通道混合召回（Hybrid Multi-Route Retrieval）
* **主流技术思想**：单纯向量检索（Dense Search）容易丢失精确的变量、函数、Bug ID 等符号；单纯关键词检索（Sparse Search）缺乏泛化和语义理解。主流方案采用 RRF（Reciprocal Rank Fusion，倒数排名融合） 算法，将两者的排名进行交叉重排。
* **QMem 架构映射**：
  * **QMem 的核心护城河**：底层完美利用了 SQLite3 的 FTS5 全文检索 与 `sqlite-vec` 向量余弦相似度 计算，通过内置的 RRF 混合检索算法进行融合，并对 Q2（全局共识）进行加权提分。配合 ONNX Runtime 搭载的 BGE-Small-ZH-v1.5，在 Mojo 原生 C 兼容编译的加加持下，完成了对“推、拉、查”三模极速召回的完美闭环。

## 🧭 第四部分：Agent 原生上下文边界与 Q2 存储方案辨析

在 QMem 落地实践中，如何处理大模型自带的纯文本规则文件（如 `CLAUDE.md` / `AGENTS.md`）与 QMem SQLite 数据库的关系，是架构设计的核心留白。

### 1. 当前架构下的干预边界
QMem 明确了外挂记忆图谱与大模型原生文本上下文的界限：
* **不主动干预 Q1 与 Q3**：
  * **Q1（工作记忆）**：完全交由 AI Agent 自身的上下文窗口处理，因其处理短时信息效率最高。
  * **Q3（静态结构）**：仅通过 `init_project_context.py` 做轻度特征提取（身份戳），不强制向 `CLAUDE.md` 写回。
* **深度接管 Q2 与 Q4**：
  * **Q4（情境记忆）**：解决 `CLAUDE.md` 无法承载海量开发历程（Bug 记录、选型推理）的痛点，全部下沉到 SQLite 并由 MCP 工具（`mem_save` / `mem_search`）按需读写。
  * **Q2（共识记忆）**：通过 `memory_promote` 机制将 Q4 中的重要经验提权，高优投喂给模型。

### 2. Q2 存放位置之争：多份散装 Markdown 文件 vs SQLite 数据库？
在之前的探讨中，关于 **第二象限（全局共识）** 到底该存在 `.md` 文档里，还是存在 SQLite 中，曾有深入辨析。特别澄清：这里的 MD 文档方案**并非指将所有东西塞进单一的 `CLAUDE.md`**，而是指按主题拆分为多份散装文档（例如 `docs/consensus/db_rules.md`, `docs/consensus/api_rules.md`）。

以下是两者的优劣态势：

#### 方案 A：通过多份散装 MD 文档存储 Q2（按主题拆分文件）
* **优势 (Pros)**：
  * **人类可读性极佳**：物理文件拆分清晰，开发者可以直接在资源管理器中分类管理、审阅和编辑。
  * **避免单文件过载**：解决了单一 `CLAUDE.md` Token 爆炸和注意力迷失的问题。
* **劣势 (Cons)**：
  * **文件名强依赖（缺乏模糊语义）**：大模型需要事先知道 `db_rules.md` 的存在，并准确推测出要阅读该文件。如果模型不知道去拉取哪个文件，或者文件名命名不够精准，这条共识就会永远沉睡（相当于人类的“文件失忆”）。
  * **上下文组装繁琐**：遇到跨域问题时，模型需要调用多次工具分别读取不同的文件。

#### 方案 B：通过 SQLite 存储 Q2（QMem 当前方案，使用 `memory_promote` 提权）
QMem 提供了 `memory_promote(obs_id)` 接口，它的底层机制是将指定记忆记录打上 `is_global=1` 标记。在后续 `mem_search` 召回时，RRF 混合搜索不仅会为其打上 1.5 倍的基础算分权重 (boost)，还会附加固定的高优排名补偿，确保其稳定排在召回列表前列。
* **优势 (Pros)**：
  * **动态语义匹配（无需猜文件名）**：基于 RRF（向量余弦相似度 + 关键词全文检索），只要模型当前任务的只言片语命中了向量空间，相关的 Q2 规则就会被精准“吸”出来。
  * **无限容量与零常驻成本**：无论存十条还是一万条共识，在未被检索前不占用任何 Token 窗口。
* **劣势 (Cons)**：
  * **依赖主动召回习惯**：如果 AI 解决问题前**忘记调用** `mem_search`，系统就无法把知识换入内存。

### 3. 特殊探讨：Q3（结构上下文）是否应该被融合进 Q2？
Q3 是客观的物理快照（例如包依赖列表、项目目录树、启动端口等），而 Q2 是主观的经验规约（例如“应该怎么写代码”）。
* **如果将 Q3 塞进 Q2（存入 SQLite 参与向量检索）**：
  * 会导致严重的**信噪比污染**。长篇的目录树和依赖列表会冲刷掉原本珍贵的“Bug避坑指南”等核心共识。
  * 向量引擎对“精确的端口号”、“特定版本号”这种纯字符匹配极度不敏感（这是 Embeddings 算法的通病）。
* **结论**：Q3 应该保持其**客观探针**的身份（如 `init_project_context.py`），独立于 Q2 经验库之外，直接在启动时静态拼装或通过独立的结构化工具提供。

### 4. 终极建议：分级分治（Hybrid Strategy）
未来的最优解并不是非黑即白，而是**分级分治的混合模式**：
1. **最高级铁律（L1 共识） → 写死在 `CLAUDE.md`**。
   * *例如：“永远不要使用 print 输出日志”、“当前正在重构 FFI 层”。* 
   * 容量控制在 500 字以内，确保 AI 绝对不会违反。
2. **领域级规约（L2 共识） → 存入 QMem SQLite (Q2 提权)**。
   * *例如：“数据库表结构演进史”、“Mojo 指针转换踩坑记录”。* 
   * 这些知识很长，且只在特定模块开发时才需要。通过在 `CLAUDE.md` 中写下一条引导规则（*“遇到复杂问题前，务必使用 mem_search 搜索 QMem 记忆库”*），让 AI 养成遇到盲区主动去 SQLite 里翻找 L2 共识的习惯。
